# 스터디 전 생각해볼 질문들

이 문서는 스터디 전에 미리 생각해보면 좋은 질문들을 모아놓은 것입니다.
답변은 각 챕터의 문서에서 확인할 수 있습니다.

---

## 1. Service와 기술 의존성

### 질문 1
> **"Service 구현체의 역할이 무엇이까요?"**

**생각해볼 점:**
- Service에서 JPA Entity를 직접 사용해도 될까?
- Service에서 Feign Client를 직접 사용해도 될까?
- Service가 기술에 의존하면 어떤 문제가 생길까?

---

## 2. Repository 인터페이스화

### 질문 2
> **"Repository도 인터페이스로 분리하나요?"**

**생각해볼 점:**
- JPA에서 JDBC로 바꾸려면 어떻게 해야 할까?
- 기술 스택을 교체할 때 코드를 얼마나 수정해야 할까?

---

## 3. Service 책임 분리

### 질문 3
> **"Service가 뚱뚱해지는 이유는 무엇일까?"**

**생각해볼 점:**
- 하나의 Service에 모든 책임이 모이는 이유는?
- 유저 조회, 토큰 생성, 쿠키 생성, DTO 변환... 모두 한 곳에 있어야 할까?

---

## 4. Orchestrator 패턴

### 질문 4
> **"여러 Service를 조율하는 계층이 필요한가?"**

**생각해볼 점:**
- Controller에서 여러 Service를 직접 호출하면 안 될까?
- Service를 재사용하려면 어떻게 해야할까?
- 비즈니스 로직과 DTO 조립을 어떻게 분리할까?

---

## 5. API 인터페이스 계층

### 질문 5
> **"Swagger 설정 때문에 Controller가 복잡하지 않나요?"**

**생각해볼 점:**
- API 스펙을 코드로 어떻게 명확하게 정의할 수 있을까?
- 다른 서비스에서 API를 호출할 때 어떻게 간단하게 API 스펙만 알 수 있을까?

---

## 6. 외부 모듈 호출

### 질문 6
> **"다른 모듈의 API를 호출할 때 Service가 Feign Client를 직접 사용해도 될까요?"**

**생각해볼 점:**
- Service에서 Feign Client를 직접 사용하면 어떤 문제가 생길까?
- Service가 기술에 의존하지 않으려면 어떻게 해야 할까?
- 모듈 간 결합을 어떻게 약하게 만들 수 있을까?

---

## 스터디 후 확인할 내용

스터디를 듣고 나서 다음을 확인해보세요:

1. ✅ Service가 비즈니스 로직만 담당하고 기술에 의존하지 않는다는 것을 이해할 수 있는가?
2. ✅ Repository 인터페이스화로 기술 스택을 유연하게 교체할 수 있는가?
3. ✅ Orchestrator 패턴으로 Service 책임을 분리할 수 있는가?
4. ✅ 외부 모듈 호출 시 Feign 패턴을 사용하여 기술 의존성을 분리할 수 있는가?
5. ✅ 도메인 모델과 Entity를 분리할 수 있는가?
