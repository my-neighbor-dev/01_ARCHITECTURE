# 1. 도입: 주제 선정 이유

많은 취업 준비생들이 이력서에 Spring Security를 활용한 인증/인가 내용을 적습니다.

**아쉬운 사례:**

```jsx
[프로젝트 A]
Trouble Shooting
- Spring Security를 활용한 인증/인가 구현
- JWT 토큰 기반 인증 시스템 구축
- Role 기반 권한 제어 구현
```
---

## 고민이 보이는 명확한 사례

**고민이 보이는 명확한 사례:**

```jsx
[프로젝트 A]
Trouble Shooting
- Role 기반 인가만으로는 다른 사용자 데이터 침범 문제가 발생해 데이터 소유권 기반 인가 정책 적용 
- AOP와 전략 패턴을 활용하여 비즈니스 로직과 보안 로직 분리
- Annotation 기반으로 반복되는 권한 체크 로직을 공통화하여 코드 중복 제거
```

---

## 핵심 메시지

**문제에 공감해야 해결이 커 보인다**

취업 준비를 할 때는 보통 '이력서에 Spring Security를 써야지'라는 도구를 먼저 정해놓고 고민하다 보니, 인증/인가 구현 외에는 활용법을 찾기 어렵습니다. 하지만 현업에서는 **'Role 기반 인가만으로는 부족하다'**는 문제가 터지면 이를 해결하기 위해 AOP와 전략 패턴을 선택합니다. 기술을 써야 할 **근거**가 자연스럽게 생기는 것이죠.

우리 스터디에서도 바로 이 **'근거'**를 만드는 연습을 함께 하려고 합니다.

---

## 오늘 스터디 내용

1. **문제 상황: Role 기반 인가의 한계**
   - "무엇을 할 수 있는가"는 체크하지만 "누구의 데이터인가"는 체크하지 못함
   - 다른 사용자의 데이터 침범 위험

2. **시도 1, 2: 단순한 접근의 한계**
   - 모든 메서드에 검증 로직 넣기 → 중복 코드
   - 공통 검증 로직 사용 → 비즈니스 로직과 보안 로직 혼재

3. **AOP + 전략 패턴으로 해결**
   - 비즈니스 로직을 건드리지 않고 보안 로직 추가
   - Annotation 기반으로 반복되는 권한 체크 공통화
   - 도메인별 다른 검증 로직을 전략 패턴으로 유연하게 처리

4. **구현 단계별 설명**
   - ResourceOwnership 인터페이스 정의
   - 전략 패턴으로 검증 로직 추상화
   - DomainFinder 인터페이스 정의
   - Annotation & Aspect 구현

5. **실제 사용 예시**
   - 강의(Lecture) 도메인 (사용자 소유권)
   - 그룹(Group) 도메인 (그룹 소유권)

---

## 스터디 목표

이 스터디를 하고 나면:

- ✅ Role 기반 인가의 한계를 이해할 수 있다
- ✅ 데이터 소유권 체크가 왜 필요한지 설명할 수 있다
- ✅ AOP를 활용하여 비즈니스 로직과 보안 로직을 분리할 수 있다
- ✅ 전략 패턴으로 도메인별 다른 검증 로직을 처리할 수 있다
- ✅ 이력서에 문제 해결 중심으로 작성할 수 있다
