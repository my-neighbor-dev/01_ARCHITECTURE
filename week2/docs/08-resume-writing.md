# 8. 이력서 작성 가이드

## 질문

> **"이 내용을 이력서에 어떻게 쓸까요?"**

> 💡 **스터디 전에 생각해보기**: [질문들 모음](./00-questions.md)을 먼저 확인해보세요!

**생각해볼 점:**
- "Redis 캐싱을 활용해 성능 20% 향상" 같은 뻔한 예시와 어떻게 다를까?
- 문제 상황을 어떻게 명확하게 표현할까?
- 기술 선택 이유를 어떻게 설명할까?

---

## 8.1 Before/After 비교

### Before: 뻔한 예시

```jsx
[프로젝트 A]
Trouble Shooting
- Redis 캐싱을 활용해 유저 데이터 조회 성능 20% 향상
- 로그인 기능 구현 시 jwt refresh token을 캐시에 저장하여 관리
```

**문제점:**
- 어떤 문제인지 명확하지 않음
- 기술 스택만 나열
- 해결 과정이 보이지 않음

---

### After: 문제 해결 중심

```jsx
[프로젝트 A]
Trouble Shooting
- 무분별한 로그인 시도(Brute-force)를 제한하기 위해 Redis Rate Limiting 적용
- Redis의 INCR과 TTL을 조합하여 Device ID당 1시간에 5회로 제한하는 로직 구현
- Device ID와 IP 주소를 함께 체크하여 다층 방어 구축
```

**장점:**
- 문제 상황 명확 (무분별한 로그인 시도)
- 해결 방법 구체적 (INCR + TTL)
- 기술 선택 이유 암시 (TTL 기능 활용)

---

## 8.2 작성 가이드

### 1. 문제 상황 명시

**좋은 예:**
- "무분별한 로그인 시도(Brute-force)를 제한하기 위해"
- "사용자의 무분별한 비밀번호 찾기 요청을 제한하기 위해"

**나쁜 예:**
- "성능 개선을 위해"
- "캐싱을 적용하기 위해"

---

### 2. 해결 방법 설명

**좋은 예:**
- "Redis의 INCR과 TTL을 조합하여"
- "Device ID당 1시간에 5회로 제한하는 로직 구현"

**나쁜 예:**
- "Redis를 사용하여"
- "Rate Limiting을 구현"

---

### 3. 기술 선택 이유

**좋은 예:**
- "TTL 기능을 활용하여 자동 만료 구현"
- "메모리 기반으로 빠른 읽기/쓰기 제공"

**나쁜 예:**
- "빠르기 때문에"
- "성능이 좋아서"

---

### 4. 구현 내용 간단히

**좋은 예:**
- "Device ID와 IP 주소를 함께 체크하여 다층 방어 구축"
- "전화번호, Device ID, IP 기반 다중 체크 로직 구현"

**나쁜 예:**
- "여러 방법으로 체크"
- "보안 강화"

## 8.4 면접 대비 질문

### 질문 1: "왜 Redis를 선택했나요?"

**답변:**
- "TTL 기능이 있어서 자동 만료가 가능했습니다"
- "메모리 기반으로 빠른 읽기/쓰기가 필요했습니다"
- "임시 데이터를 저장하는 것이 목적이었기 때문에 DB보다 캐시가 적합했습니다"

---

### 질문 2: "Rate Limiting을 어떻게 구현했나요?"

**답변:**
- "Redis의 INCR 명령어로 횟수를 증가시키고, 첫 요청 시 TTL을 설정했습니다"
- "Device ID, 전화번호, IP 주소를 각각 체크하여 다층 방어를 구축했습니다"
- "1시간에 5번을 초과하면 예외를 발생시켜 차단했습니다"

---

### 질문 3: "왜 Device ID와 IP를 함께 사용했나요?"

**답변:**
- "Device ID만으로는 쿠키 삭제 후 재시도가 가능했습니다"
- "IP만으로는 공유 IP 환경에서 정상 사용자도 영향을 받을 수 있었습니다"
- "둘을 함께 사용하여 더 강력한 보안을 제공했습니다"
