# 1. 도입: 주제 선정 이유

많은 취업 준비생들이 이력서에 Redis를 적고 싶어 합니다. 하지만 대부분 다음과 같이 작성합니다:

**아쉬운 사례:**

```jsx
[프로젝트 A]
Trouble Shooting
- Redis 캐싱을 활용해 유저 데이터 조회 성능 20% 향상
- 로그인 기능 구현 시 jwt refresh token을 캐시에 저장하여 관리
```

- *작성자의 시선:* "수치를 넣어서 한 일을 명확하게 표현해야겠다."
- *현직자의 시선:* "어떤게 Trouble인걸까?, 캐싱을 쓰면 성능이 개선되는건 당연한데"

**'무슨 문제가 있었는지'** 보다 **'무엇을 사용했는지'**가 먼저 나오기 때문에, 면접관 입장에서는 Redis를 도입해야하는 Trouble(문제상황)에 대해 공감하지 못하고, 이로 인해 Shooting(문제해결)에 임팩트를 느끼지 못합니다.

---

## 고민이 보이는 명확한 사례

**고민이 보이는 명확한 사례:**

```jsx
[프로젝트 A]
Trouble Shooting
- 무분별한 로그인 시도(Brute-force)를 제한하기 위해 Redis Rate Limiting 적용
- Redis의 INCR과 TTL을 조합하여 Device ID당 1시간에 5회로 제한하는 로직 구현
```

- *현직자의 시선:* "단순히 '성능이 좋아졌다'고 주장하는 게 아니라, **'악성 유저의 공격'** 같은 구체적인 상황을 미리 대비했네"

이력서에 Redis를 적더라도 위와 같이 작성하면 훨씬 더 **'문제 해결 중심적인'** 개발자로 보입니다.

사실 내용이 어렵게 느껴질 수 있지만, `INCR`이나 `TTL` 설정은 3분만 검색해도 누구나 사용할 수 있습니다. Redis로 공격을 막는 로직 역시 1시간이면 구현 가능하죠. 

**결국 중요한 건 '구현 기술'보다 '어떤 문제를 풀 것인가'를 정의하는 힘입니다.**

---

## 핵심 메시지

**문제에 공감해야 해결이 커 보인다**

취업 준비를 할 때는 보통 '이력서에 Redis를 써야지'라는 도구를 먼저 정해놓고 고민하다 보니, 성능 개선 외에는 활용법을 찾기 어렵습니다. 하지만 현업에서는 **'무분별한 로그인 시도 방지'** 같은 문제가 터지면 이를 해결하기 위해 Redis를 선택합니다. 기술을 써야 할 **근거**가 자연스럽게 생기는 것이죠.

우리 스터디에서도 바로 이 **'근거'**를 만드는 연습을 함께 하려고 합니다.

---

## 오늘 스터디 내용

1. **토큰을 쿠키에 저장하는 이유** (week1 개선)
   - Response Body에 토큰을 보내는 것의 문제점
   - XSS 공격 방지를 위한 쿠키 사용

2. **문제 해결 과정: 종이와 펜으로 생각하기**
   - 공격자가 옆에 있다고 가정하고 해결 방법 생각하기
   - 단계별로 최적화하는 과정

3. **Redis Rate Limiting으로 무분별한 로그인 시도 방지**
   - 캐시의 특성(TTL, 빠른 읽기/쓰기) 활용
   - Device ID와 IP 기반 다층 방어

4. **브라우저 vs 앱에서의 쿠키 처리**
   - 각 환경에서의 쿠키 처리 방법

이 스터디를 하고 나면:

- ✅ 토큰을 쿠키에 저장하는 이유를 이해할 수 있다
- ✅ 문제 해결 과정을 단계별로 생각할 수 있다
- ✅ 캐시의 특성을 활용해서 문제를 해결할 수 있다
- ✅ Redis Rate Limiting을 구현할 수 있다
- ✅ 이력서에 문제 해결 중심으로 작성할 수 있다
