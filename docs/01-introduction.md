# 1. 도입: 질문으로 시작하기

## 질문

> **"Service 구현체의 역할이 무엇일까요?"**

> 💡 **스터디 전에 생각해보기**: [질문들 모음](./00-questions.md)을 먼저 확인해보세요!

**생각해볼 점:**
- Service에서 JPA Entity를 직접 사용해도 될까?
- Service에서 Feign Client를 직접 사용해도 될까?
- Service가 기술에 의존하면 어떤 문제가 생길까?

## 일반적인 문제

- Service에서 JPA Entity를 직접 사용
- Service에서 Feign Client를 직접 사용
- Service가 기술에 의존하게 됨

## 하지만 올바른 방법은?

**오늘 스터디에서 알아볼 내용:**

1. **Service는 비즈니스 로직만 담당** - 기술에 의존하지 않음
2. **Repository 인터페이스화** - 기술 스택을 유연하게 교체
3. **Orchestrator 패턴** - Service 책임 분리 및 DTO 조립
4. **외부 모듈 호출** - Feign 패턴으로 기술 의존성 분리

---

## 스터디 목표

이 스터디를 하고 나면:

- ✅ Service는 비즈니스 로직만 담당하고 기술에 의존하지 않는다는 것을 이해할 수 있다
- ✅ Repository 인터페이스로 기술 스택을 유연하게 교체할 수 있다
- ✅ Orchestrator 패턴으로 Service를 얇게 유지하고 책임을 분리할 수 있다
- ✅ 외부 모듈 호출 시 Feign 패턴을 사용하여 기술 의존성을 분리할 수 있다

---

## 예제 프로젝트 소개

**강의 사이트** 프로젝트를 예제로 사용합니다.

- **User 모듈**: 유저 정보 조회 기능
- **Auth 모듈**: 로그인 기능 (토큰 생성, 쿠키 설정)

**핵심**: 모듈 간 의존성을 어떻게 관리할 것인가?
